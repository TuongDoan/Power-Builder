# Reference 17: Code Architecture (Three-Layer Pattern)

Use this reference when structuring the app's source code. This is the **mandatory architecture** for all Code App projects.

## Overview

Every Code App follows a strict three-layer architecture with clear separation of concerns:


**Core principle**: Props down, events up. Components receive data via props and emit events via callbacks. No component ever calls a service directly.

---

## Layer 1: Presentation (`src/components/`)

Components are **pure and presentational**. They:
- Receive data via props
- Emit events via callback props
- Contain **no** service calls and **no** business logic
- Manage only minor, local UI state (e.g. a controlled input value before submission)

```typescript
// вң… Good вҖ” pure component
<RecordList
  records={records}      // data down
  loading={loading}      // data down
  onEdit={startEdit}     // event up
  onDelete={deleteRecord} // event up
/>
```

### Recommended components
| Component | Responsibility |
|-----------|----------------|
| `Header` | App title, version badge |
| `ErrorMessage` | Conditional error banner |
| `RecordList` | Grid/table of cards + "New" button |
| `RecordCard` | Single record display with Edit/Delete |
| `RecordForm` | Create/edit form with lookup dropdowns |
| `Footer` | Informational text |

Always create a barrel `index.ts` in `src/components/` to re-export all components.

---

## Layer 2: Business Logic (`src/hooks/`)

Custom hooks encapsulate all state and async operations. They are the **only** place that calls generated services.

### Mandatory hooks for CRUD apps
| Hook | Responsibility |
|------|----------------|
| `useRecords` | CRUD operations, form state (create/edit/cancel), error handling |
| `useRelatedTable` | Loads related table data for dropdown/lookup fields |
| `useLookupResolver` | Resolves lookup GUIDs to display names on-demand |

### Hook interface example
```typescript
const {
  records,           // Record[]
  loading,           // boolean
  error,             // string | null
  selectedRecord,    // Record | null
  isCreating,        // boolean
  startCreate,       // () => void
  startEdit,         // (record: Record) => void
  cancelForm,        // () => void
  handleFormSubmit,  // (data: FormData) => Promise<void>
  deleteRecord,      // (id: string) => Promise<void>
} = useRecords();
```

Always create a barrel `index.ts` in `src/hooks/` to re-export all hooks.

---

## Layer 3: Data Access (`src/generated/`)

Auto-generated by `pac code add-data-source`. **Never edit these files manually** вҖ” regenerate if the schema changes.

Each generated data source provides:
- `src/generated/models/<Table>Model.ts` вҖ” TypeScript entity types
- `src/generated/services/<Table>Service.ts` вҖ” CRUD service (`create`, `get`, `getAll`, `update`, `delete`)
- `.power/schemas/dataverse/<table>.Schema.json` вҖ” Dataverse schema

Each service provides: `create()`, `get()`, `getAll()`, `update()`, `delete()`, `getMetadata()`.

## Error Handling Strategy

Three-level strategy is mandatory:

```
Service Level   вҶ’ Generated services return IOperationResult with .error (create/update/get)
                  OR return void (delete)
    вҶ“
Hook Level      вҶ’ Check result.error for create/update, try-catch for delete + user-friendly message
    вҶ“
Component Level вҶ’ ErrorMessage component renders the string from state
```

```typescript
// Hook level вҖ” create/update: check result.error
const result = await RecordsService.create(payload);
if (result.error) {
  setError(`Failed to create: ${result.error.message}`);
  return;
}
// Access created record via result.data (NOT result.value)
const newRecord = result.data;
```

```typescript
// Hook level вҖ” delete returns void, use try-catch
try {
  await RecordsService.delete(id);
} catch (err: unknown) {
  const msg = err instanceof Error ? err.message : 'Unknown error';
  setError(`Failed to delete record: ${msg}`);
  console.error('Delete error:', err);
}
```

> [!CAUTION]
> **IOperationResult shape:** `create()`, `update()`, `get()`, and `getAll()` return `IOperationResult<T>` with `.data` (the records) and optional `.error`. **Always use `result.data`** вҖ” `.value` does not exist.
> **`delete()` returns `void`** вҖ” no result object. Use try-catch for error handling.
> **Boolean fields:** Pass actual `true`/`false` вҖ” never `0`/`1`. See `references/30-connect-dataverse-pac.md` for details.
> **Virtual name fields:** Do NOT include `<lookupname>name` in `select` arrays. See `references/16-lookup-patterns.md`.

## Performance Patterns

### Query optimization (mandatory)
Always use `select`, `top`, and `orderBy` to limit data transfer:

```typescript
RecordsService.getAll({
  select: ['primaryid', 'name', '_lookupfield_value'],
  orderBy: ['createdon desc'],
  top: 50,
});
```

### Lookup resolution
See `references/16-lookup-patterns.md` for the full on-demand lookup resolution pattern. Key rule: use individual `Service.get(guid, { select: [...] })` calls rather than loading entire tables.

### Conditional rendering
Form components should only mount when needed (create/edit mode). Keep the initial render lightweight.

## Layout integrity (mandatory)

If the user later asks for mobile support, add breakpoints as an **additive** step вҖ” never remove or compromise the desktop layout.

### Full Width & Multi-Column Layout
Since desktop support is the default, the app should maximize screen real estate:
- **Full Width**: The main container should span the full width of the browser window (100% width).
- **Multi-Column**: Use a 2+ column layout for data-rich views (e.g., side navigation + content, or list on left + details on right) to efficiently use horizontal space.
- **Avoid Narrow Containers**: Do not constrain the main content to a narrow centered column unless specifically requested for a landing page or simple form.

### Layout integrity checklist
After every UI creation or modification, verify all of the following:

| Check | What to look for |
|---|---|
| **Spacing** | Consistent margins, padding, and gaps. No cramped or excessively loose sections. |
| **Alignment** | Elements align to a clear grid. No unexpected shifts, off-center text, or misaligned columns. |
| **Container sizing** | No horizontal overflow, no unintended scrollbars, no content clipping. |
| **Typography** | Consistent font sizes, weights, and line heights across all sections. Headings follow a clear hierarchy. |
| **Interactive targets** | Buttons, inputs, and dropdowns are comfortably clickable (minimum 36 px hit area). Adequate spacing between adjacent interactive elements. |
| **Visual weight** | Important actions (primary buttons, CTAs) are visually prominent. Secondary/tertiary actions are visually subordinate. |
| **Empty states** | All lists, tables, and data sections have graceful empty-state rendering. |
| **Scrolling** | Only the intended regions scroll. Fixed headers/sidebars remain stable during scroll. |

Flag and fix any layout issues **before** proceeding to the next phase or deploying.

## Best Practices Checklist

- [ ] Component composition вҖ” complex UIs from simple, focused pieces
- [ ] Custom hooks for logic separation and reuse
- [ ] TypeScript throughout вҖ” generated models, typed props, typed hook returns
- [ ] Props down, events up вҖ” unidirectional data flow
- [ ] Loading and error states at every async boundary
- [ ] Generated services only вҖ” no raw `fetch` or `axios` calls
- [ ] Query optimization вҖ” `select`, `top`, `orderBy` on every `getAll()` call
- [ ] `result.error` check after every `create()` and `update()` call
- [ ] Use `result.data` (NOT `result.value`) to access records from `IOperationResult`
- [ ] `delete()` returns `void` вҖ” use try/catch, not `result.error`
- [ ] Boolean columns: pass `true`/`false`, NEVER `0`/`1` (even if TS enum suggests it)
- [ ] Virtual `*name` fields (e.g. `categoryidname`): NEVER include in `select`, they come automatically
- [ ] `create()` payload: use `as unknown as Omit<Base, 'id'>` to bypass `ownerid` requirement
- [ ] Efficient lookup resolution вҖ” individual `get()` via `useLookupResolver`
